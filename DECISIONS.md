# DECISIONS.md

## 2026-04-22
### 决策：GPS 启动定位短时重复发送，超时后使用默认经纬度兜底
**为什么**
- UI 只需要系统启动时获得设备经纬度，不需要 GPS 线程长期持续发送。
- 启动阶段采用 0.5 秒短时重复发送，可降低 UDP 单包丢失导致 UI 未收到位置的风险。
- UM980 冷启动或室内环境可能长时间无有效定位；如果一直等待真实 fix，UI 端会迟迟拿不到设备位置。
- 设备部署点通常是固定的，因此 `gps.py` 中的默认经纬度可以作为硬件异常、无卫星或天线问题时的可靠兜底。

**影响**
- `main_tracking_v9.py` 启动后创建 `gps_sender_thread`。
- GPS 线程最多等待 `GPS_FIX_TIMEOUT_SECONDS = 60` 秒获取真实 GGA 定位。
- 若获取成功，发送真实经纬度给 UI。
- 若超时、无卫星、无有效 GGA、串口异常或打开失败，发送 `gps.py` 中的 `DEFAULT_LATITUDE` / `DEFAULT_LONGITUDE`。
- GPS 包按 `GPS_UI_SEND_DURATION = 0.5`、`GPS_UI_SEND_INTERVAL = 0.1` 短时重复发送，发送完成后线程退出。
- 新增 UI GPS 包格式：`0x03 + float(latitude) + float(longitude)`，实现为 `struct.pack('!Bff', 0x03, latitude, longitude)`。

**验证**
- `gps.py` 可直接运行调试 UM980 NMEA 输出。
- `udp_ui_receiver.py` 可模拟 UI 端监听 UDP `9999`，并解析 `0x03` GPS 包。
- 本地已验证 `0x03` 包长度为 9 字节，首字节为 `3`，后续为纬度和经度两个 float。

## 2026-04-22
### 决策：GPS 串口纳入现有跨平台串口配置体系
**为什么**
- 项目已有云台、激光、IMU 的平台默认串口和环境变量覆盖机制，GPS 应复用同一配置风格，避免硬编码散落在业务逻辑里。
- Linux 下 `/dev/ttyUSB*` 会受 USB 插拔顺序影响，必须允许通过环境变量覆盖到稳定设备节点。

**影响**
- 新增 `GPS_PORT = _serial_port("GPS_PORT", "gps")`。
- 当前默认值：
  - Windows: `COM8`
  - Linux: `/dev/ttyUSB2`
- `GPS_PORT` 可通过环境变量覆盖；后续部署建议改用 `/dev/serial/by-id/...`。
- `gps.py` 也支持读取 `GPS_PORT` 环境变量，保持单独调试脚本和主程序一致。

**验证**
- `main_tracking_v9.py` 导入后可正确解析 GPS 默认端口。
- Linux 日志中已能看到 GPS 线程按配置端口启动，例如 `waiting for one valid fix on /dev/ttyUSB2@115200`。

## 2026-04-16
### 决策：云台抢占改为“同目标按轴更新，切目标整条替换”
**为什么**
- 旧逻辑即使已经拆成 `AZ_PREEMPT_DEG` / `EL_PREEMPT_DEG`，一旦任一轴触发抢占，仍会把整条 `Az/El` 命令一起替换。
- 这会导致低方位阈值在同一目标连续跟踪时“挟持”俯仰轴，使俯仰自己的抢占阈值失去意义。
- 但若目标身份发生切换，又不能只替换单轴，否则会出现“新方位 + 旧俯仰”的混合指向。

**影响**
- 当前抢占阈值正式拆分为：
  - `AZ_PREEMPT_DEG = 0.5`
  - `EL_PREEMPT_DEG = 0.8`
- 控制线程更新规则变为：
  1. `new_track_id != active_track_id` 时，整条替换 `Az/El`
  2. `new_track_id == active_track_id` 时，按轴判断并只更新满足阈值的轴
- 抢占日志新增：
  - `mode=track_switch`
  - `mode=axis_update`
  - `axes=Az/El/Az+El`

**验证**
- 代码层面已完成修改并通过语法检查。
- 这次决策的核心验证目标不是“抢占次数变多或变少”，而是：
  - 同目标连续跟踪时，方位误差不再因为整条命令跳变而周期性重新放大
  - 目标切换时，不出现混合指向
- 最新实机回归结果尚待补充到本条目下。

## 2026-04-14
### 决策：室内出现“持续无有效激光距离”时，优先检查目标条件而不是立即怀疑主流程故障
**为什么**
- 在 Linux 接收端 `10.72.2.28:8888` 的重复 `linear` 联调中，`main_tracking_v9.py` 已明确打印 `[Init] Real laser enabled on /dev/ttyUSB1`、`[LaserThread]` 和 `[Laser] start_measurement mode=continuous`，说明激光初始化与后台线程都已启动。
- 但 `logs/main_tracking_v9_20260414_161730.log` 中，尽管每次 `GimbalSettled` 后都尝试取激光缓存，整场测试仍只有 `[Laser] No valid laser distance, use mono distance`。
- 随后仅将激光指向更远目标，不改代码、不改串口配置，`logs/main_tracking_v9_20260414_162745.log` 就恢复了连续真实测距，说明问题主要受目标距离/反射条件影响。

**影响**
- 激光联调排障顺序调整为：
  1. 先确认激光线程、连续测量命令和 settled 触发都存在
  2. 再检查目标距离、材质、反射面角度与瞄准方向
  3. 最后才怀疑主流程代码或串口协议回归
- 保持当前“连续测量 + settled 时消费最新缓存值”的结构不变。
- 保持当前“无有效激光值时退回 mono distance”的降级策略不变。

**验证**
- `logs/main_tracking_v9_20260414_161730.log`：
  - 激光线程已启动，连续测量命令已发送
  - 整场测试中 settled 后均退回 mono distance
- `logs/main_tracking_v9_20260414_162745.log`：
  - 在更远目标条件下恢复 `GimbalSettled=22`
  - `GimbalTimeout=0`
  - `Laser Triggered=22`
  - `Laser No valid=0`
  - 真实激光距离约 `6.2m ~ 7.9m`

## 2026-04-14
### 决策：真实激光接入主流程，并保留 `USE_MOCK_LASER` 作为统一切换开关
**为什么**
- 原先主程序虽然有 `LASER_PORT`、`SharedHardwareState.raw_laser_dist` 和 `sddm_laser.py`，但没有任何后台线程把真实激光数据写入共享状态，导致主流程实际一直退回 mono distance。
- 调试阶段既需要真实激光联调，也需要在无激光硬件时保留可重复的模拟输入，因此需要像云台一样提供统一开关。

**影响**
- `main_tracking_v9.py` 新增 `USE_MOCK_LASER`
- `USE_MOCK_LASER=False` 时：
  - 启动真实 `SDDMLaser`
  - 后台线程发送连续测量命令
  - 持续读取并缓存真实激光值
- `USE_MOCK_LASER=True` 时：
  - 不打开真实激光串口
  - 使用当前主目标的新鲜 mono 距离模拟激光输入

**为什么选择连续测量**
- 当前控制结构是在 `GimbalSettled` 后消费“最新激光缓存值”，而不是在 settled 当下发单次测量并阻塞等待。
- 连续测量更适合现有非阻塞串口/控制线程结构，也更容易保持日志与目标绑定关系稳定。

**验证**
- 短时测试中曾出现“仅无效帧”的情况，主流程正确退回 mono distance。
- 长时 `linear` 场景测试中，真实云台 `COM3` + 真实激光 `COM8` 已验证：
  - `GimbalSettled=19`
  - `GimbalTimeout=0`
  - `Laser Triggered=19`
  - `Laser No valid=0`
- 真实激光返回的是实物环境距离 `4.7m ~ 6.5m`，说明真实激光链路已生效；发送脚本中的 `320m` 仅作用于视觉输入。

## 2026-04-14
### 决策：按手册补强 SDDM 激光驱动帧校验，但保持驱动接口不变
**为什么**
- 原始驱动的起停测量命令与 CRC 思路基本正确，但数据读取时只做 CRC 校验，不足以区分“CRC 恰好正确但不是目标数据帧”的情况。
- 该项目是硬件耦合系统，驱动接口应稳定，优先补强校验而不是改写上层调用方式。

**影响**
- `sddm_laser.py` 现在额外校验：
  - 数据帧 `MsgCode=0x03`
  - `PayloadLen=0x04`
- 无效标志位不再返回负距离，统一按“无有效测距”处理
- 保持 `start_measurement()`、`stop_measurement()`、`read_distance()` 接口不变

**验证**
- `COM8` 原始串口抓包曾出现连续的 `fb0300040000000002`，被正确识别为“有回包但无有效距离”
- 后续长测中已出现持续有效测距，说明校验补强没有破坏正常数据解析

## 2026-04-10
### 决策：使卡尔曼滤波器时间步长 (dt) 严格依赖实际帧率
**为什么**：测试表明，硬编码的 `MIN_DT=0.08` 与实际的 15 FPS (~0.066s) 不匹配。这会导致卡尔曼滤波器系统性地低估目标的实际角速度，从而使前馈补偿过于保守，云台跟随滞后。
**影响**：
- 废弃固定的时间步长下限。
- 卡尔曼预测必须基于两次更新之间的实际时间差或帧率期望。
**已实施**：
- `main_tracking_v9.py` 中动态参数返回的 `MIN_DT` 已改为 `0.001`。
- `StandardKalmanTrack` 默认 `self.min_dt` 已改为 `0.001`。
- 该值只用于防止异常极小 `dt`，不再把 15FPS 场景的 `~0.066s` 抬高到 `0.08s`。
**待验证**：
- 复测 `udp_sender_tracking_scenarios.py` 的 `linear` 场景：320m、5.1m/s、15FPS、`cx=700 -> 3686.67`、`cy=1080`。
- 目标角速度估算应从旧日志中的约 `0.76°/s` 更接近按观测角度反推的 `~0.91°/s`。

## 2026-04-10
### 决策：解耦方位轴 (Pan) 与俯仰轴 (Pitch) 的动态控制参数
**为什么**：实测发现俯仰轴的机械响应显著慢于方位轴（从 -9° 爬升至 28.5° 需要超过 2s）。使用单一全局的 `PREDICT_DELAY=0.25` 过于粗糙，无法同时兼顾两个轴的不同物理迟滞。
**影响**：
- 拆分提前量控制，分别实现 `AZ_PREDICT_DELAY` 和 `EL_PREDICT_DELAY`。
- 未来的控制逻辑修改必须尊重多轴动力学的不对称性。
**已实施的临时缓解**：
- 当前尚未拆分 `PREDICT_DELAY`，仍保留单一 `PREDICT_DELAY=0.25`。
- 已新增启动位姿初始化：系统启动后先通过 `gimbal_cmd_queue` 投递 `cmd_id=0, track_id=-1` 的初始化命令，目标姿态为 `Az=GIMBAL_AZ_BASE`、`El=0.0°`。
- 该初始化命令仍由 `gimbal_control_thread` 统一下发，保持云台串口单一控制线程所有权。
**待验证**：
- 上电前确认 `Az=GIMBAL_AZ_BASE`、`El=0.0°` 方向安全无遮挡，不会撞限位。
- 复测首个目标出现时俯仰轴是否不再从约 `-9°` 大幅追赶到 `28.5°`。

## 2026-04-10
### 决策：分离“首次捕获 (Initial Acquisition)”与“稳定跟踪 (Stable Tracking)”状态
**为什么**：单一参数无法兼顾全周期。目标刚进入视野时需要激进的参数以实现“快速拉近”；一旦进入视场中心区域，则需要平滑的参数来“抑制抖动”。
**影响**：
- 在主控逻辑中引入阶段标志 (Phase State Machine)。
- 针对捕获阶段和稳定阶段，应用不同的前馈权重、卡尔曼增益或限速策略。

## 2026-04-10
### 决策：强制拉开“抢占阈值 (Preempt)”与“到位阈值 (Settle)”的迟滞区间 (Hysteresis)
**为什么**：当 `PREEMPT=0.7` 和 `SETTLE_THRESHOLD=0.8` 过于接近时，云台极易在“即将到位”和“被新微小误差打断”的边缘状态下反复横跳，导致严重的机械震荡。
**影响**：
- 必须确立严格的迟滞控制原则，例如 `PREEMPT_DEG` 应显著大于 `SETTLE_THRESHOLD`。
- 在状态到位之前，必须过滤掉落在死区内的微小位置更新指令。
**已实施**：
- `GIMBAL_PREEMPT_DEG` 已从 `0.7` 调整为 `1.5`。
- `GIMBAL_SETTLE_THRESHOLD` 已从 `0.8` 先收紧到 `0.5`，随后根据云台约 `0.1°` 精度进一步收紧到 `0.3`。
**待验证**：
- 观察旧日志中 `delta=0.71~0.74°` 的抢占是否消失或明显减少。
- 观察更严格 `0.3°` 到位阈值是否导致 `GimbalTimeout` 增多；若超时明显增多，再考虑在 `0.3~0.5°` 区间回调。
